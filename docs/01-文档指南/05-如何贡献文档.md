# 如何贡献文档
- **知识传承**：通过文档沉淀项目经验，便于新成员快速上手和团队知识共享

本文档旨在指导开发者和贡献者如何为SutWxApp项目贡献和维护文档。良好的文档对于项目的可持续发展至关重要，我们欢迎所有形式的文档贡献。

## 1. 文档贡献的意义

- **提升项目质量**：清晰的文档有助于用户更快理解和使用项目
- **降低维护成本**：完善的文档减少重复问题和支持请求
- **促进社区参与**：友好的文档鼓励更多开发者参与贡献
- **项目专业度体现**：高质量文档是项目成熟度的重要指标

## 2. 文档类型与范围

项目文档主要包括以下类型：

- **用户指南**：面向终端用户的操作说明
- **管理员指南**：WordPress管理员使用文档
- **开发者指南**：开发人员技术文档、API文档等
- **安装部署文档**：环境配置和安装步骤
- **功能模块说明**：各功能模块的详细说明
- **常见问题解答**：FAQ和故障排除指南

## 3. 文档贡献流程

### 3.1 准备工作

1. **了解项目**：通过阅读现有文档和README了解项目基本情况
2. **克隆代码库**：
   ```bash
   git clone https://github.com/sutchan/SutWxApp.git
   cd SutWxApp
   ```
3. **创建分支**：
   ```bash
   git checkout -b docs/your-docs-name
   ```

### 3.2 编写文档

1. **选择文档位置**：根据文档类型将文档放在合适的目录下
2. **遵循文档格式**：按照[文档格式规范](./00-文档格式规范.md)编写
3. **参考模板**：使用[文档模板](./04-文档模板.md)确保结构一致性
4. **添加必要内容**：
   - 清晰的标题和小标题
   - 详细的步骤说明
   - 代码示例（如有）
   - 相关图片（如有）
   - 版本信息和最后更新时间

### 3.3 提交文档

1. **添加文件**：
   ```bash
   git add docs/your-docs-path/your-docs-file.md
   ```
2. **提交更改**：
   ```bash
   git commit -m "docs: 添加/更新XXX文档"
   ```
3. **推送分支**：
   ```bash
   git push origin docs/your-docs-name
   ```
4. **创建Pull Request**：在GitHub上创建Pull Request，描述文档变更内容

## 4. 文档格式规范

请严格遵循以下文档格式规范：

### 4.1 基本格式

- **文件格式**：使用Markdown格式（.md文件）
- **编码格式**：UTF-8 无BOM
- **换行符**：使用Unix LF格式
- **语言**：中文文档为主，关键技术术语使用英文

### 4.2 标题规范

- 使用#、##、###等Markdown标题标记
- 标题层级要清晰，避免跳跃式层级
- 文档标题使用一级标题（#）
- 章节标题使用二级标题（##）
- 子章节使用三级标题（###）

### 4.3 内容规范

- **代码块**：使用```bash```、```javascript```等语法高亮
- **表格**：使用Markdown表格语法，保持对齐
- **图片**：使用相对路径，添加alt文本
- **链接**：内部链接使用相对路径，外部链接使用完整URL
- **列表**：使用有序或无序列表时保持格式一致

### 4.4 语言风格

- **简明扼要**：用简洁的语言表达准确的信息
- **专业术语**：使用统一的技术术语
- **一致的人称**：使用第二人称（"您"）或第三人称，保持一致
- **避免缩写**：除非是广泛使用的缩写，否则首次出现时提供全称

## 5. 文档审核与发布

### 5.1 审核流程

1. **自动化检查**：系统会自动检查文档格式和编码
2. **人工审核**：项目维护者会审核文档内容的准确性和完整性
3. **反馈修改**：如有需要，审核者会提供修改建议
4. **合并发布**：审核通过后，文档将被合并到主分支

### 5.2 审核标准

- **内容准确性**：文档内容必须与实际代码和功能一致
- **格式规范性**：遵循项目的文档格式规范
- **表达清晰度**：语言表达清晰，易于理解
- **信息完整性**：包含必要的背景信息、步骤说明和注意事项

## 6. 文档维护

### 6.1 定期更新

- 当代码或功能发生变化时，相应的文档需要及时更新
- 每个版本发布前，检查并更新相关文档
- 定期审查文档，修复错误或过时信息

### 6.2 版本同步

- 文档版本应与代码版本保持一致
- 在文档开头标明当前版本号和最后更新时间
- 重要的功能变更需要在文档中特别说明

## 7. 常见问题

### 7.1 文档放在哪个目录？

根据文档类型选择合适的目录：
- 项目概述：`docs/00-项目概览/`
- 管理员指南：`docs/02-管理员指南/`
- 开发者指南：`docs/03-开发者指南/`
- 用户指南：`docs/04-用户指南/`
- 功能模块说明：`docs/05-功能模块/`

### 7.2 如何添加代码示例？

使用带语法高亮的代码块：

```javascript
// 示例代码
function example() {
  console.log('Hello World');
}
```

### 7.3 如何处理多语言文档？

- 主要维护中文文档
- 重要文档可以提供英文版本，使用`-en_US`后缀区分
- 确保中英文文档内容同步更新

### 7.4 文档提交后多久会被审核？

通常情况下，文档PR会在1-3个工作日内完成审核。如有紧急需求，请在PR描述中说明。

## 8. 最佳实践

- **先阅读现有文档**：避免重复工作或与现有文档冲突
- **从用户角度思考**：考虑不同用户群体的需求和理解能力
- **保持简洁明了**：避免冗长复杂的描述
- **提供实际示例**：使用真实的代码示例和操作步骤
- **定期审查更新**：随着项目发展及时更新文档
