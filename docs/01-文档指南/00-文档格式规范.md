# 文档格式规范

**版本号：1.0.16**
*最后更新时间：2024年11月1日*

## 1. 目的与范围

本文档定义了SutWxApp项目文档的标准格式和风格规范，旨在确保所有项目文档具有一致的格式、清晰的结构和专业的外观，提高文档的可读性和可维护性。

本规范适用于项目中的所有文档，包括但不限于：
- 用户指南
- 管理员指南
- 开发者指南
- API文档
- 安装部署文档
- 功能模块说明文档

## 2. 文件格式与命名规范

### 2.1 文件格式

- **文件类型**：使用Markdown格式（.md扩展名）
- **编码格式**：UTF-8 无BOM
- **换行符**：使用Unix LF格式

### 2.2 文件命名

- **命名规则**：使用中文或英文，避免混用
- **数字前缀**：文档使用数字前缀进行排序，格式为"XX-文档标题.md"，例如：
  - `01-项目简介.md`
  - `02-安装指南.md`
- **英文文档**：英文文档使用`-en_US`后缀，例如：`01-项目简介-en_US.md`
- **避免使用**：空格、特殊字符（除连字符-和下划线_外）

## 3. 文档结构规范

### 3.1 标题层级

- **一级标题**：使用`#`，仅用于文档主标题
- **二级标题**：使用`##`，用于主要章节
- **三级标题**：使用`###`，用于子章节
- **四级标题**：使用`####`，用于更详细的子部分
- **标题层级**：最多使用四级标题，避免过深的层级嵌套

示例：

```markdown
# 主标题

## 第一章

### 1.1 子章节

#### 1.1.1 详细部分
```

### 3.2 文档开头

所有文档必须包含以下信息作为开头：

```markdown
# 文档标题

**版本号：1.0.16**
*最后更新时间：2024年11月1日*
```

### 3.3 文档结尾

所有文档必须在结尾处添加分隔线和版本信息：

```markdown
---

**版本号：1.0.16**
*最后更新时间：2024年11月1日*
```

## 4. Markdown语法规范

### 4.1 文本格式

- **加粗**：使用`**文本**`而非`__文本__`
- **斜体**：使用`*文本*`而非`_文本_`
- **代码**：行内代码使用`` `代码` ``
- **删除线**：使用`~~删除的文本~~`

### 4.2 列表

- **无序列表**：使用连字符`-`作为标记
- **有序列表**：使用数字加句点`1.`
- **缩进**：子列表缩进2个空格
- **空行**：列表项与上下文之间空一行

示例：

```markdown
- 无序列表项1
  - 子列表项1.1
  - 子列表项1.2
- 无序列表项2

1. 有序列表项1
2. 有序列表项2
```

### 4.3 代码块

- **代码块**：使用三个反引号```，并指定语言类型
- **语言标记**：必须为代码块添加语言标记以启用语法高亮
- **说明文字**：代码块前可添加简短说明
- **行号**：不推荐添加行号，除非有特殊需要

示例：

```javascript
// 这是一段JavaScript代码示例
function hello() {
  console.log('Hello World');
}
```

### 4.4 表格

- **表格格式**：使用标准Markdown表格语法
- **对齐**：使用冒号指定对齐方式（左对齐、居中、右对齐）
- **内容长度**：避免表格内容过长，考虑将复杂内容拆分
- **空行**：表格与上下文之间空一行

示例：

```markdown
| 姓名 | 年龄 | 职位 |
|------|------|------|
| 张三 | 30   | 开发者 |
| 李四 | 25   | 设计师 |
```

### 4.5 链接

- **内部链接**：使用相对路径，例如`[链接文本](./文档路径.md)`
- **外部链接**：使用完整URL，例如`[GitHub](https://github.com)`
- **链接文本**：使用描述性文本，避免使用"点击这里"
- **锚点链接**：对于长文档，可使用锚点链接跳转到特定部分

示例：

```markdown
[API文档](./03-开发者指南/03-API接口文档.md)
[GitHub仓库](https://github.com/sutchan/SutWxApp)
[返回顶部](#主标题)
```

### 4.6 图片

- **图片路径**：使用相对路径
- **Alt文本**：必须提供描述性的alt文本
- **图片说明**：重要图片可在下方添加说明
- **图片大小**：大型图片应适当调整大小

示例：

```markdown
![Logo](./images/logo.png)
*项目Logo*
```

### 4.7 引用

- **引用块**：使用`>`添加引用
- **嵌套引用**：使用`>>`添加嵌套引用
- **引用内容**：保持引用内容的清晰和简洁

示例：

```markdown
> 这是一段引用文本
>> 这是嵌套引用
```

## 5. 内容风格规范

### 5.1 语言使用

- **主要语言**：使用中文编写文档
- **技术术语**：使用行业标准的技术术语
- **英文术语**：英文术语和代码保持英文
- **标点符号**：中文文档使用全角标点，代码和英文中使用半角标点

### 5.2 表达方式

- **简洁明了**：使用简洁的语言表达准确的信息
- **清晰结构化**：内容层次分明，重点突出
- **一致的人称**：使用第二人称（"您"）或第三人称，保持一致
- **主动语态**：优先使用主动语态，避免被动语态
- **避免缩写**：除非是广泛使用的缩写，否则首次出现时提供全称

### 5.3 文档内容

- **准确性**：确保文档内容与实际代码和功能一致
- **完整性**：包含必要的背景信息、步骤说明和注意事项
- **实用性**：提供实用的指导和示例，避免空洞的描述
- **时效性**：定期更新文档，确保内容不过时

## 6. 特殊文档类型的额外规范

### 6.1 API文档

- **接口结构**：清晰列出请求方法、URL、参数、响应格式
- **参数说明**：详细描述每个参数的类型、必选/可选、默认值和用途
- **响应示例**：提供成功和失败的响应示例
- **错误码**：列出可能的错误码及其含义

### 6.2 安装部署文档

- **环境要求**：明确列出硬件和软件要求
- **步骤说明**：提供详细的、按顺序的安装步骤
- **配置示例**：提供配置文件的示例和说明
- **验证方法**：说明如何验证安装是否成功

### 6.3 用户指南

- **步骤清晰**：操作步骤简单明了，使用编号列出
- **截图辅助**：添加必要的截图辅助理解
- **常见问题**：包含常见问题和解决方法
- **术语解释**：对专业术语进行解释

## 7. 文档审核标准

文档提交审核时，将按照以下标准进行检查：

- **格式合规**：符合本规范中的格式要求
- **内容准确**：信息准确无误，与实际功能一致
- **结构清晰**：文档结构合理，层次分明
- **语言规范**：语言表达清晰，没有语法错误
- **版本一致**：文档版本号与代码版本保持一致
- **链接有效**：所有链接都能正常访问

## 8. 版本控制

### 8.1 版本号

- **版本号格式**：遵循语义化版本规范（SemVer）：MAJOR.MINOR.PATCH
- **版本同步**：文档版本应与代码版本保持一致
- **版本更新**：每次修改文档后更新版本号和最后更新时间

### 8.2 版本历史

重要文档应包含版本历史记录，格式如下：

```markdown
| 版本号 | 修改日期 | 修改内容 | 修改人 |
|-------|---------|---------|-------|
| 1.0.16 | 2024-11-01 | 初始版本 | Sut |
| 1.0.17 | 2024-12-01 | 更新安装步骤 | Sut |
```

## 9. 最佳实践

- **先构思再编写**：先确定文档结构和主要内容，再进行详细编写
- **参考现有文档**：参考其他高质量文档的格式和风格
- **保持一致性**：在整个项目中保持一致的文档风格
- **简洁原则**：内容简洁明了，避免冗余
- **定期审查**：定期审查和更新文档，确保内容不过时
