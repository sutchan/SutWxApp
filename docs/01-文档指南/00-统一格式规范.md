# 统一格式规范

## 1. 概述

本文档定义了SutWxApp项目所有文档必须遵循的统一格式规范，确保文档的视觉呈现一致性、可读性和专业性。

## 2. 文件基本规范

### 2.1 文件格式

- **文件类型**：使用Markdown格式（.md扩展名）
- **编码格式**：UTF-8 无BOM
- **换行符**：使用Unix LF格式
- **语言**：中文文档为主，关键技术术语可使用英文

### 2.2 文件命名

- **命名规则**：`XX-文档名称.md`
- **编号**：使用两位数字序号，从00开始，按文档重要性和阅读顺序排列
- **名称**：使用中文，清晰表达文档内容，避免使用缩写或模糊术语
- **英文文档**：英文文档使用`-en_US`后缀，例如：`01-项目简介-en_US.md`
- **避免使用**：空格、特殊字符（除连字符-和下划线_外）

### 2.3 头部注释

所有文档必须包含标准的头部注释，格式如下：

```markdown
<!--
文件名: XX-文档名称.md
版本号: MAJOR.MINOR.PATCH
更新日期: YYYY-MM-DD
作者: 作者姓名
描述: 文档的简短描述
-->
```

## 3. 文档结构规范

### 3.1 标题层级

- **一级标题**：使用`#`，仅用于文档主标题
- **二级标题**：使用`##`，用于主要章节
- **三级标题**：使用`###`，用于子章节
- **四级标题**：使用`####`，用于更详细的子部分
- **标题层级**：最多使用四级标题，避免过深的层级嵌套
- **标题格式**：标题文字前无空格，标题文字后无标点符号

### 3.2 章节编号

文档章节采用层次化编号，格式如下：

```markdown
## 1. 一级章节

### 1.1 二级章节

#### 1.1.1 三级章节
```

### 3.3 文档内容结构

每个文档应包含以下基本结构：

1. **头部注释**：标准的文件头部注释
2. **主标题**：文档的正式名称（一级标题）
3. **版本信息**：当前版本号和最后更新时间
4. **目录**：自动生成的文档目录（可选）
5. **正文内容**：按照逻辑分节组织的文档内容
6. **版本历史**：文档的版本变更记录
7. **页脚信息**：文档的版本号和最后更新时间

## 4. 文本格式规范

### 4.1 基本文本格式

- **字体**：默认字体（系统或阅读器默认）
- **字号**：默认字号（系统或阅读器默认）
- **行间距**：1.5倍行间距
- **段落间距**：段落之间空一行
- **缩进**：正文不缩进，列表和引用除外

### 4.2 强调文本

- **加粗**：使用`**文本**`，用于重要内容
- **斜体**：使用`*文本*`，用于强调或术语
- **代码**：行内代码使用`` `代码` ``
- **删除线**：使用`~~删除的文本~~`，用于过时内容

### 4.3 列表

- **无序列表**：使用连字符`-`作为标记
- **有序列表**：使用数字加句点`1.`
- **缩进**：子列表缩进2个空格
- **空行**：列表项与上下文之间空一行
- **列表项格式**：列表项后空一格再开始文本

### 4.4 代码块

- **代码块**：使用三个反引号```，并指定语言类型
- **语言标记**：必须为代码块添加语言标记以启用语法高亮
- **说明文字**：代码块前可添加简短说明
- **行号**：不推荐添加行号，除非有特殊需要
- **代码块与上下文**：代码块与上下文之间空一行

### 4.5 表格

- **表格格式**：使用标准Markdown表格语法
- **对齐**：使用冒号指定对齐方式（左对齐、居中、右对齐）
- **内容长度**：避免表格内容过长，考虑将复杂内容拆分
- **空行**：表格与上下文之间空一行
- **表头**：必须包含表头行

### 4.6 链接

- **内部链接**：使用相对路径，例如`[链接文本](./文档路径.md)`
- **外部链接**：使用完整URL，例如`[GitHub](https://github.com)`
- **链接文本**：使用描述性文本，避免使用"点击这里"
- **锚点链接**：对于长文档，可使用锚点链接跳转到特定部分

### 4.7 图片

- **图片路径**：使用相对路径
- **Alt文本**：必须提供描述性的alt文本
- **图片说明**：重要图片可在下方添加说明
- **图片大小**：大型图片应适当调整大小
- **图片与上下文**：图片与上下文之间空一行

### 4.8 引用

- **引用块**：使用`>`添加引用
- **嵌套引用**：使用`>>`添加嵌套引用
- **引用内容**：保持引用内容的清晰和简洁
- **引用与上下文**：引用块与上下文之间空一行

## 5. 内容风格规范

### 5.1 语言使用

- **主要语言**：使用中文编写文档
- **技术术语**：使用行业标准的技术术语
- **英文术语**：英文术语和代码保持英文
- **标点符号**：中文文档使用全角标点，代码和英文中使用半角标点

### 5.2 表达方式

- **简洁明了**：使用简洁的语言表达准确的信息
- **清晰结构化**：内容层次分明，重点突出
- **一致的人称**：使用第二人称（"您"）或第三人称，保持一致
- **主动语态**：优先使用主动语态，避免被动语态
- **避免缩写**：除非是广泛使用的缩写，否则首次出现时提供全称

### 5.3 文档内容

- **准确性**：确保文档内容与实际代码和功能一致
- **完整性**：包含必要的背景信息、步骤说明和注意事项
- **实用性**：提供实用的指导和示例，避免空洞的描述
- **时效性**：定期更新文档，确保内容不过时

## 6. 版本信息规范

### 6.1 版本号格式

遵循语义化版本规范（SemVer），版本号格式为：

```
MAJOR.MINOR.PATCH
```

- **MAJOR**：不兼容的破坏性变更
- **MINOR**：向下兼容的新功能
- **PATCH**：向下兼容的问题修复

### 6.2 版本信息位置

- **文档头部**：在主标题下方必须标注当前版本号和更新时间
- **格式**：
  ```markdown
  **版本号：MAJOR.MINOR.PATCH**
  *最后更新时间：YYYY年MM月DD日*
  ```

### 6.3 版本历史记录

每个文档必须包含版本历史记录，格式如下：

```markdown
## 版本历史

| 版本号 | 更新日期 | 更新内容 | 更新人 |
|-------|---------|---------|-------|
| 1.0.0 | 2025-11-24 | 初始版本 | Sut |
| 1.0.1 | 2025-11-25 | 更新内容说明 | 作者姓名 |
```

- **版本顺序**：按照版本号从新到旧的顺序排列
- **更新内容**：简要描述主要的更新内容
- **更新人**：负责文档更新的人员姓名

## 7. 特殊文档类型规范

### 7.1 API文档

- **接口结构**：清晰列出请求方法、URL、参数、响应格式
- **参数说明**：详细描述每个参数的类型、必选/可选、默认值和用途
- **响应示例**：提供成功和失败的响应示例
- **错误码**：列出可能的错误码及其含义

### 7.2 安装部署文档

- **环境要求**：明确列出硬件和软件要求
- **步骤说明**：提供详细的、按顺序的安装步骤
- **配置示例**：提供配置文件的示例和说明
- **验证方法**：说明如何验证安装是否成功

### 7.3 用户指南

- **步骤清晰**：操作步骤简单明了，使用编号列出
- **截图辅助**：添加必要的截图辅助理解
- **常见问题**：包含常见问题和解决方法
- **术语解释**：对专业术语进行解释

## 8. 审核标准

文档提交审核时，将按照以下标准进行检查：

- **格式合规**：符合本规范中的格式要求
- **内容准确**：信息准确无误，与实际功能一致
- **结构清晰**：文档结构合理，层次分明
- **语言规范**：语言表达清晰，没有语法错误
- **版本一致**：文档版本号与代码版本保持一致
- **链接有效**：所有链接都能正常访问

## 9. 最佳实践

- **先构思再编写**：先确定文档结构和主要内容，再进行详细编写
- **参考现有文档**：参考其他高质量文档的格式和风格
- **保持一致性**：在整个项目中保持一致的文档风格
- **简洁原则**：内容简洁明了，避免冗余
- **定期审查**：定期审查和更新文档，确保内容不过时
- **使用模板**：使用标准文档模板确保结构一致性
